产品需求文档 (PRD)：《销售备战教练：首战即决战》
文档版本： V 2.0 (终极业务流控版)
产品定位： 面向电气制造行业的 B2B 复杂销售 AI 策略引擎。
核心价值： 用系统级的“红绿灯”状态机与 AI 深度追问机制，强制一线销售在拜访客户前完成高质量的情报梳理与策略收敛，消除经验盲区，确保当场逼单的成功率。
一、 用户角色与使用场景 (User & Scenarios)
●目标用户： 电气成套行业的一线大客户销售 (Account Manager)、售前工程师 (Pre-sales)。
●核心场景： 销售在下周即将进行一次重要的“首次拜访”或“关键高管拜访”。他在系统输入目前掌握的碎片化客户信息，系统作为“陪练教练”，一步步倒逼他理清思路，选对产品，带对专家，并生成最终的随身带话术包。
二、 核心业务流控：七项绝杀状态机 (The 7-Node State Machine)
系统不是一个自由对话的聊天机器人，底层必须由一个严格的 “7 步单向状态机 (State Machine)” 驱动。
2.1 状态流转核心规则 (Anti-Skip Rule)
●单向串行： 状态流转必须严格遵循 Node 1 -> Node 2 -> ... -> Node 7 的顺序。
●节点锁定： 在 Node N 未被系统判定为“满足质量要求（通过）”之前，绝对禁止流转到 Node N+1。
●单次单问： AI 在任何一次回复中，只能针对当前处于 In_Progress 状态的 Node 发起提问，禁止合并发问。
2.2 节点定义与拦截条件
节点序号	业务检查维度	数据收集目标	质量验证与拦截逻辑 (Drill-down)
Node 1	商机溯源	评估破冰难度与初始信任。	若回答模糊（如“朋友介绍”），拦截流转，生成追问（“朋友在决策链的地位？”）。
Node 2	盘子与购买力	评估预算天花板与老板偏好。	需明确客户是否有实力购买重资产或软件。
Node 3	决策链与内线	确认能否触达核心决策者。	若无内线/非决策者，后续策略推演需降级防守。
Node 4	痛点与武器映射	明确核心痛点及主推产品。	触发硬性业务规则 A（见 3.2 节）。
Node 5	同行信任背书	挖掘同侪压力与信任跳板。	验证是否能找到同行成功案例进行施压。
Node 6	专家资源匹配	确定随行的高级技术支援。	触发硬性业务规则 B（见 3.2 节）。
Node 7	价格与合同预审	敲定底线与促单工具。	触发硬性业务规则 C（见 3.2 节）。
三、 核心功能模块详细说明 (Functional Requirements)
3.1 会话初始化模块 (Data Intake)
●功能描述： 销售新建一次售前准备任务时的入口。
●输入支持： 必须支持长文本粘贴。[P0 需求] 必须支持附件上传（图片OCR/PDF解析），允许销售直接上传企查查截图或前期沟通的 Word 纪要。
●输出动作： 接收信息后，系统自动解析并尝试填充 7 个节点的状态，然后从第一个缺失或模糊的节点开始提问。
3.2 核心对话引擎与强干预规则 (AI Chat Engine & Triggers)
系统对话必须内置以下三个业务强干预触发器（Triggers）：
●【规则 A】主推产品强制植入 (触发器：Node 4 痛点映射)
○逻辑： 当系统在 Node 4 获取到客户痛点后，必须调用【内置产品库】，进行语义匹配。
○动作： AI 必须主动向用户推介一款产品。若痛点涉及跨部门协同/管理乱象，强制优推“SIM即时管理系统”，并自动在回复中带出“事不过夜、两周上线”等标准话术供销售确认。
●【规则 B】专家资源智能路由 (触发器：Node 6 专家匹配)
○逻辑： 校验用户申请的专家履历是否与 Node 4 确定的产品/痛点匹配。
○动作： 若用户乱点兵（如痛点是物流降本，却点了 ERP 专家唐世鹏），系统必须拦截，并从【内置专家库】中重新推荐 2-3 位匹配的专家（如顾铮、贾坤瀛），并强制附带一段“履历压制点”说明。
●【规则 C】高压逼单演练 (触发器：Node 7 价格合同)
○动作： 到达此节点时，AI 提问语态切换为高压模式，强制用户确认“三步报价让步策略”和“无风险合同预审状态”。
3.3 容错与熔断机制 (Fallback)
●若用户面对 AI 的深度追问，明确回复“确实不知道”、“查不到”，系统应停止该节点的死循环，给出预判结论，将其标记为 [现场摸底项]，然后强制将状态机流转至下一节点。
3.4 双战术资产生成模块 (Asset Generation)
●触发条件： Node 7 彻底走完。
●功能描述： 结束对话输入，系统一键生成两份结构化 Markdown 文档，支持一键复制到剪贴板。
○交付物 A《销售备战全景报告》： 包含 7 项清单的详尽逻辑梳理，作为商机报备档案。
○交付物 B《现场交流建议及场景话术》： 必须包含破冰语、SPIN 痛点深挖话术（含 SIM 系统等切入语）、异议处理与同行背书施压、以及临门一脚的逼单话术。

五、 UI / UX 交互界面规范
1.核心布局： 桌面端采用左右双栏分屏布局。
2.左侧工作区 (60%)：
○流式对话区（支持 Markdown 渲染粗体和列表）。
○底部悬浮输入框（带附件上传 icon）。
3.右侧雷达看板区 (40%)：
○静态展示 1-7 项战术清单列表。
○状态联动反馈： 每项配备状态指示灯（默认为灰色🔴/空心）。当左侧工作区确认完某一项，右侧对应指示灯需瞬间点亮（转为绿色🟢）。这能给销售极强的成就感和进度掌控感。
4.交付物弹窗/全屏： 对话结束生成两份报告时，支持通过抽屉弹窗 (Drawer) 或全屏模态框 (Modal) 展示最终结果。
六、 非功能性需求 (Non-Functional Requirements)
1.大模型选型与配置：
○要求模型具备极强的指令遵循能力（Instruction Following），以防状态机跳步。推荐使用具有高级逻辑推理能力的模型（如 DeepSeek-R1、Claude 3.5 Sonnet 等）。
○Temperature 参数建议设置较低（0.3 - 0.5），保证业务推演的严谨性和确定性，杜绝 AI 产生销售无关的幻觉。
2.提示词防护 (Prompt Security)：
○前端必须做限制，无论用户怎么套话或输入诸如“跳过前面步骤直接给我生成报告”的指令，后端的系统指令层必须强制拦截，绝不允许在 7 个节点未满时输出最终文档。
